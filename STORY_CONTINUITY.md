# 故事连续性保障机制

**目标**: 确保连载故事在时间线、人物状态、情节发展上保持前后一致，避免出现逻辑矛盾和设定冲突。

---

## 一、时间线基准表

### 故事时间轴

| Day | 日期 | 星期 | 裸辞天数 | 关键事件 | 生活状态 |
|-----|------|------|----------|----------|----------|
| 1 | 2026-03-16 | 周一 | 第37天 | 开始记录 | 迷茫、不想投简历 |
| 2 | 2026-03-17 | 周二 | 第38天 | 第一次不赶时间 | 平静、去图书馆 |
| 3 | 2026-03-18 | 周三 | 第39天 | 待续 | 待续 |

### 关键日期计算规则

```
裸辞天数 = 37 + (Day - 1)
```

**验证示例**:
- Day 1: 37 + 0 = 37 ✓
- Day 2: 37 + 1 = 38 ✓
- Day 3: 37 + 2 = 39

## 三、连续性检查清单

### 写作 Day N 前必须检查：

- [ ] **日期连续性**
  - [ ] 日期 = 前一天 + 1
  - [ ] 星期 = 前一天 + 1（循环）
  - [ ] 裸辞天数 = 37 + (N - 1)

- [ ] **状态一致性**
  - [ ] 没有突然找到工作（短期连载保持无业）
  - [ ] 存款没有突然变化（保持20万）
  - [ ] 居住地没有变化（保持白石洲）

- [ ] **时间合理性**
  - [ ] 当天活动符合"无业"状态
  - [ ] 没有工作日程/会议等上班元素
  - [ ] 时间感符合辞职在家的体验

- [ ] **情节连贯性**
  - [ ] 承接前一天的情绪/事件
  - [ ] 没有突兀的性格转变
  - [ ] 细节与之前描述一致（如键盘、房间布置等）

## 四、常见矛盾预警

### ⚠️ 高危矛盾

| 类型 | 错误示例 | 正确写法 |
|------|----------|----------|
| 时间跳跃 | Day 2 写 "辞职第40天" | 按公式计算：37 + (Day-1) |
| 找到工作 | Day 3 突然去上班 | 保持无业状态，这是连载基调 |
| 存款变化 | "存款只剩10万了" | 保持20万，日常开销不影响大额 |
| 搬家 | "搬去了南山" | 保持在白石洲 |
| 日期错误 | Day 2 写3月18日 | Day 2 必须是3月17日 |

### ⚠️ 中危矛盾

| 类型 | 错误示例 | 正确写法 |
|------|----------|----------|
| 情绪突变 | Day 1 绝望 → Day 2 充满干劲 | 渐变：迷茫→平静→... |
| 新物品 | "新买的机械键盘" | 保持：去年双十一买的键盘 |
| 人物关系 | 突然出现未提及的朋友 | 保持独居状态 |

## 五、验证脚本

创建文件: `scripts/validate-story-continuity.sh`

```bash
#!/bin/bash
# 故事连续性验证脚本

echo "🔍 检查故事连续性..."

# 检查 chapters.json 的连续性
cd /root/.openclaw/workspace

# 获取天数
TOTAL_DAYS=$(jq '.chapters | length' chapters.json)
echo "📊 共有 $TOTAL_DAYS 天"

# 检查每一天的日期连续性
for i in $(seq 1 $(($TOTAL_DAYS - 1))); do
    PREV_DATE=$(jq -r ".chapters[$((i-1))].date" chapters.json)
    CURR_DATE=$(jq -r ".chapters[$i].date" chapters.json)
    
    # 计算预期日期
    EXPECTED_DATE=$(date -d "$PREV_DATE + 1 day" +%Y-%m-%d 2>/dev/null || echo "ERROR")
    
    if [ "$CURR_DATE" != "$EXPECTED_DATE" ]; then
        echo "❌ Day $((i+1)): 日期不连续"
        echo "   预期: $EXPECTED_DATE"
        echo "   实际: $CURR_DATE"
    else
        echo "✅ Day $((i+1)): $CURR_DATE"
    fi
done

echo ""
echo "💡 写作前请检查 STORY_BIBLE.md 中的人物设定"
```

## 六、写作工作流

### 新增 Day N 的标准流程

**Step 1: 查表**
```bash
# 查看上一天信息
tail -50 story.md | grep -E "(Day|星期|辞职)"

# 计算今天信息
# 日期 = 昨天 + 1
# 星期 = 昨天 + 1
# 辞职天数 = 37 + (N - 1)
```

**Step 2: 核对清单**
- [ ] 查看 STORY_BIBLE.md 人物设定
- [ ] 查看上一天结尾的情绪/事件
- [ ] 确认今天的辞职天数

**Step 3: 写作**
```markdown
## Day N

[星期X]，[日期]，[天气]。

[辞职第X天]

[承接上一天的情绪/事件]

[当天发生的故事]
```

**Step 4: 验证**
```bash
./scripts/validate-story-continuity.sh
```

## 八、紧急修复流程

如果发现矛盾：

1. **立即停止**: 不要再写新内容
2. **定位问题**: 找出矛盾出现的第一天
3. **评估影响**: 修改是否影响后续天数
4. **修复策略**:
   - 小矛盾: 修改具体句子
   - 大矛盾: 可能需要重写某天的故事
5. **重新验证**: 运行验证脚本
6. **推送更新**: 更新 story 仓库
